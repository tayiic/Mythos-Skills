## 二、需要补充的方法论
### 必须加的 (3 个) 🔴 M7: 计划驱动开发方法论 (Plan-Driven Development)
为什么必须：这是 superpowers 最核心的差异点。

superpowers 的 writing-plans 是社区评价最高的 skill：

- "Transforms vague ideas into actionable steps with exact file paths"
- 每个 task 2-5 分钟可完成
- 假设执行者"零上下文、品味差、不爱写测试"
我们的版本应该更极致：

- 不是教 AI 写计划，而是教人类/AI 什么是一个好计划的标准
- 结合 P1-P8 原则：计划本身也是"上下文"，需要简洁、渐进式披露
- 输出格式标准化：计划文档 = 另一种 SKILL.md（≤55行骨架 + details 按需） 🟡 M8: 代码审查方法论 (Code Review Framework)
为什么必须：没有审查 = TDD 白做。

superpowers 的 requesting-code-review 在每个 task 之间自动触发。但它的审查标准是隐式的。

我们应该显式化：

- 审查什么？（不是风格，是正确性 + 安全性 + 可测试性）
- 用什么标准审？（P1-P8 同样适用于审查代码质量）
- 谁来审？（AI 自审 Inline Self-Review vs 人工审查）
- 审查产出是什么？（APPROVED / REQUEST_CHANGES + 具体理由） 🟢 M9: 验证三角方法论 (Verification Triad)
为什么重要：防止"我觉得修好了"综合征。

当前我们的 verify-triple-jump 在 harness 里但没有独立为方法论：

- 单元测试通过 ≠ 功能正确
- 功能正确 ≠ 集成无问题
- 集成无问题 ≠ 生产安全
三跳验证：

```
Unit Test Pass → Integration Smoke → Production Sanity Check
     ↓                ↓                      ↓
  pytest ok      e2e runner             staging deploy test
```
### 可选加的 (2 个) ⚪ M10: 上下文工程方法论 (Context Engineering)
Karpathy 的思想已经融入 P2 (渐进式披露)，但可以独立为一套方法：

- SCOFI (Single Context of Focus and Intent) — 一次只聚焦一个意图
- Token 预算管理 — 如何在有限窗口内最大化信息密度
- 优先级排序 — 什么信息必须在 SKILL.md，什么可以放 references/ ⚪ M11: 子代理编排方法论 (Subagent Orchestration)
superpowers 的 subagent-driven-development 是其最独特的功能：

- 每个 task 启动一个全新子代理
- 子代理之间零上下文污染
- 主代理在 task 之间做 code review
我们可以提炼出通用的子代理编排模式。

## 三、需要多少 Skills？
### 数量策略：精 vs 博
策略 数量 代表 Stars 我们的选择 极精 (单点突破) 1-3 graphify (30k⭐) 单一极致功能 ❌ 太少 精 ( curated ) 7-10 superpowers (160k⭐) 14 skills 但卖的是方法论 ✅ 最佳 博 (大而全) 50-150 ECC (166k⭐) 151 skills ❌ 维护成本高 全量注册表 5000+ awesome-openclaw (47k⭐) 列表型 ❌ 不是我们的定位

结论：7-10 个原创 Skills + gallery 无限。

理由：

- superpowers 用 14 个 skills 拿到 160k ⭐，证明 数量不需要多
- 我们的差异化在于 每个 skill 都严格遵循 P1-P8 ，质量 > 数量
- 7-10 个足以覆盖完整开发生命周期
- gallery 负责展示"优化能力"（数量不限）
